# Detection – INC-009 SSH Brute Force

## 1. Basic Kibana Filter (Lab Validation)

Data view: `filebeat-*`

**Step 1 – Confirm failures exist:**
```kql
message : "Failed password for"
```

**Step 2 – Confirm successes exist:**
```kql
message : "Accepted password for"
```

**Step 3 – Confirm key-based logins:**
```kql
message : "Accepted publickey for"
```

**Step 4 – Scope to specific host:**
```kql
host.name : "ubuntu-victim" and message : "Failed password for"
```

**Step 5 – Scope to specific attacker IP:**
```kql
host.name : "ubuntu-victim" and message : "172.16.0.11" and message : "Failed password for"
```

Time range: set to **Last 15 minutes** to cover the active attack window and observe the spike generated by the `hydra` run.

---

## 2. Broad SSH Pattern (All Systems)

Goal: detect all SSH authentication events (failures + successes) across all hosts in the lab.

```kql
message : ("Failed password for" or "Accepted password for" or "Accepted publickey for")
```

> ⚠️ This query returned **~948 events** in the lab — valid for discovery and hunting but too noisy for a detection rule. Use the scoped queries above for alerting.

## 3. Detection Rule 1 – SSH Failed Authentication (All Systems)

**Rule type:** Custom query  
**Index pattern:** `filebeat-*`

**Query:**
```kql
message : ("Failed password for" or "Failed publickey for" or "Invalid user")
```

**Rule settings:**

| Field | Value |
|-------|-------|
| Name | `Linux SSH Failed Authentication Attempt` |
| Description | Detects failed SSH password or key-based authentication attempts in Linux auth logs ingested via Filebeat. |
| Severity | Medium |
| Risk score | 47 |
| MITRE Tactic | TA0006 – Credential Access |
| MITRE Technique | T1110 – Brute Force |
| Schedule | Every 1 minute |
| Look-back | 1 minute |
| Tags | `linux`, `ssh`, `bruteforce`, `auth`, `lab` |

## 5. Why EQL Sequence Did Not Work in Discover

During testing, the following EQL sequence query was attempted in the Kibana Discover search bar:

```
sequence by host.name, source.ip, user.name with maxspan=5m
  [authentication where event.outcome == "failure"]
  [authentication where event.outcome == "success"]
```

This produced a parse error: `Expected ":", "<", "<=", ">", ">=", AND, OR, end of input but """ found.`

**Root cause:** EQL `sequence` syntax is not valid in the Discover search bar (which only accepts KQL/Lucene). EQL must be run via:
- **Dev Tools:** `GET filebeat-*/_eql/search { ... }`
- **Elastic Security → Rules → Event Correlation (EQL)** rule type

Additionally, EQL sequence detection requires ECS-mapped fields (`source.ip`, `user.name`, `event.outcome`, `event.category`) which are not yet parsed in the current raw Filebeat data.

## 6. EQL Upgrade Path (Future)

Once ECS fields are parsed, this sequence rule can be used in the Elastic Security Event Correlation rule type:

```eql
sequence by host.name, source.ip, user.name with maxspan=5m
  [authentication where event.outcome == "failure"]
  [authentication where event.outcome == "success"]
```

Required ingest pipeline grok patterns:
```
Failed password for %{USER:user.name} from %{IP:source.ip} port %{NUMBER:source.port:int}
Accepted password for %{USER:user.name} from %{IP:source.ip} port %{NUMBER:source.port:int}
Accepted publickey for %{USER:user.name} from %{IP:source.ip} port %{NUMBER:source.port:int}
Invalid user %{USER:user.name} from %{IP:source.ip} port %{NUMBER:source.port:int}
```

Plus setting `event.category: authentication` and `event.outcome: failure/success` on each document.
